コミット前にコードの品質をレビューしてください。

## 手順

### Step 1: 差分の取得

1. `git diff` と `git diff --cached` で未コミットの変更を確認
2. 変更がない場合はその旨を伝えて終了
3. 差分が大きい場合（10ファイル以上）はファイル単位で確認する

### Step 2: アーキテクチャ・責務の分離

CLAUDE.md のアーキテクチャルールに基づき、以下を検査する：

- **層の依存関係**: コンポーネント → カスタムフック → `lib/api.ts` → バックエンド の流れになっているか。コンポーネントから直接 `fetch` や `lib/api.ts` を呼び出していないか
- **責務の混在**: ビジネスロジックや API 呼び出しがコンポーネント内に書かれていないか。表示ロジックとデータ取得が適切に分離されているか
- **Server / Client Component の使い分け**: インタラクション不要のコンポーネントに不要な `"use client"` が付与されていないか。逆に `useState`/`useEffect`/イベントハンドラを使うのに `"use client"` が抜けていないか
- **API 呼び出し経路**: API 呼び出しが必ず `lib/api.ts`（openapi-fetch）経由になっているか
- **生成コードの直接編集**: `lib/generated/` 以下を直接編集していないか
- **状態管理の方針**: 選択中の銘柄・期間が URL searchParams で管理されているか。サーバーデータが SWR でキャッシュ管理されているか

### Step 3: 命名規則

TypeScript / Next.js / React の命名慣例とプロジェクト固有のルールを検査する：

- **コンポーネント**: PascalCase（例: `CandleChart`, `WatchlistPanel`）。ファイル名もコンポーネントと一致させる
- **カスタムフック**: `use` 接頭辞 + camelCase（例: `useCandles`, `useWatchlist`）
- **変数・関数**: camelCase（例: `fetchCandles`, `selectedSymbol`）
- **型・インターフェース**: PascalCase（例: `Candle`, `WatchlistItem`）。`I` 接頭辞は付けない
- **定数**: 用途に応じて。モジュールレベルの不変値は SCREAMING_SNAKE_CASE（例: `DEFAULT_INTERVAL`）、ローカルの const は camelCase
- **ファイル名**:
  - コンポーネント: PascalCase（例: `CandleChart.tsx`）
  - フック・ユーティリティ・ロジック: camelCase（例: `useCandles.ts`, `indicators.ts`）
  - App Router の特殊ファイル: `page.tsx`, `layout.tsx`, `loading.tsx` など Next.js 規約に従う
- **Boolean**: `is`/`has`/`should` などの接頭辞（例: `isLoading`, `hasError`）
- **イベントハンドラ**: `handle` 接頭辞（例: `handleClick`）、props 側は `on` 接頭辞（例: `onSelect`）

### Step 4: コード品質

以下の観点で品質を検査する：

- **型安全性**: `any` の使用を避け、`unknown` + 絞り込み、または適切な型を使っているか。型アサーション（`as`）の濫用がないか
- **null / undefined**: オプショナルチェイニング（`?.`）や Nullish 合体（`??`）が適切に使われているか
- **エラーハンドリング**: `try/catch` で握り潰していないか、SWR/フェッチのエラーが UI に伝播されているか
- **不要な再レンダリング**: `useEffect` の依存配列が正しいか。`useMemo`/`useCallback` の濫用または欠落
- **副作用の配置**: `useEffect` 内でのデータ取得は SWR/カスタムフックに置き換えられないか
- **コードの重複**: 同じ処理が複数箇所に書かれていないか（共通化を検討）
- **コンポーネントの長さ**: 1つのコンポーネントが長すぎないか（目安: 200行以上は分割を検討）
- **props の数**: props が多すぎないか（目安: 7個以上はオブジェクトへの集約を検討）
- **マジックナンバー / 文字列**: 意味のある定数として切り出されているか
- **キー属性**: リストレンダリングで `key` に index ではなく安定した一意な値を使っているか
- **アクセシビリティ**: ボタン/リンクの使い分け、`aria-*` 属性、画像の `alt` 属性が適切か
- **環境変数**: ハードコードせず `NEXT_PUBLIC_*` などの命名規則に従っているか
- **未使用のインポート / 変数**: 残っていないか

### Step 5: テスト品質（テストファイルが含まれる場合）

- **テストの粒度**: ユーザーから見える振る舞いを検証しているか（実装詳細に依存しすぎていないか）
- **Testing Library の利用**: `getByRole` / `getByLabelText` など、ユーザー視点のクエリを優先しているか（`getByTestId` の濫用を避ける）
- **ケースの網羅性**: 正常系・異常系・ローディング状態・空状態がカバーされているか
- **モックの適切さ**: API モック（MSW など）やフックのモックが適切に使われているか
- **`describe` / `it` の命名**: 日本語または英語で意図が明確に書かれているか
- **非同期処理**: `findBy*` / `waitFor` が適切に使われ、`act` 警告が出ないか

### Step 6: レビュー結果の出力

以下のフォーマットで結果を出力する。問題がない場合もその旨を明記する：

```text
## レビュー結果

対象: <N>ファイル (+<追加行数>, -<削除行数>)

### 問題点（要修正）
問題がある場合、ファイル名と行番号を添えて具体的に指摘する。

### 改善提案（任意）
必須ではないが、コード品質向上のための提案があれば記載する。

### 良い点
良いコードがあれば積極的に言及する。
```

## 注意事項

- レビューは日本語で行う
- 指摘には必ずファイル名と該当箇所を含める
- 修正案がある場合は具体的なコード例を示す
- 過度に細かい指摘（スタイルの好みレベル）は避け、実質的な品質改善に焦点を当てる
- このスキルはコードの変更は行わない（レビューのみ）
